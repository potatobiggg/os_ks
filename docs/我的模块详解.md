## 1. 磁盘布局与核心数据结构

### 1.1 宿主文件 = 逻辑块数组

整个文件系统持久化在一个宿主文件（默认 `filesystem.dat`）中。该文件被划分为 **546 个固定大小的逻辑块**，每块 512 字节。`persist.c` 通过 `fseek + fread/fwrite` 将块号映射为文件偏移：

```
offset = blkno × BLOCK_SIZE   // BLOCK_SIZE = 512
```

| 块号范围 | 用途 | 大小 |
|---------|------|------|
| 0 | 超级块（SuperBlock） | 512 B |
| 1 ~ 33 | inode 区（32 块，每块 16 个 inode = 共 512 个 inode） | 16 KB |
| 34 ~ 545 | 数据区（512 块） | 256 KB |

### 1.2 关键数据结构

```c
// 磁盘 inode — 每个文件/目录的元数据（32 字节）
typedef struct {
    uint16_t di_mode;           // 高 4 位 = 文件类型（FT_REGULAR/FT_DIR），低 9 位 = 权限
    uint16_t di_nlink;          // 硬链接计数（目录至少为 2，因为有 . 和 ..）
    uint16_t di_uid, di_gid;    // 所有者 UID / GID
    uint32_t di_size;           // 文件长度（字节）
    uint16_t di_addr[10];       // 混合索引：
                                //   [0..7] 直接索引（8 个块 → 最大 4 KB 文件）
                                //   [8]    单间接索引
                                //   [9]    双间接索引
} DiskInode;

// 超级块 — 文件系统元数据的根（512 字节，占块 0）
typedef struct {
    uint16_t s_nfree;               // 空闲 inode 栈的当前高度
    uint16_t s_free_inode[50];      // 空闲 inode 号栈（LIFO）
    uint16_t s_nfree_block;         // 空闲 data block 栈的当前高度
    uint16_t s_free_block[50];      // 空闲 data block 号栈（LIFO）
    uint16_t s_fmod;                // 脏标记：0=clean, 1=需要写回磁盘
    uint16_t s_inode_area_start;    // inode 区起始块号
    uint16_t s_inode_area_size;     // inode 区块数
    uint16_t s_data_area_start;     // 数据区起始块号
    uint16_t s_data_area_size;      // 数据区块数
    uint16_t s_total_blocks;        // 总块数
    uint16_t s_pad[148];            // 填充到 512 字节
} SuperBlock;

// 内存 inode — 磁盘 inode 的内存缓存
typedef struct Minode {
    DiskInode mi_dinode;            // 磁盘 inode 的内存副本
    uint32_t  mi_ino;               // 该 inode 的编号
    uint16_t  mi_count;             // 引用计数（>0 表示正在使用，不宜回收）
    uint16_t  mi_flag;              // 标志位（ILOCKED/IUPD/IACC）
    struct Minode *mi_next, *mi_prev; // 哈希桶双向链表
} MInode;

// 目录项 — 目录数据块的内容就是 DirEntry 数组
typedef struct {
    char     de_name[14];           // 文件名（null-terminated）
    uint16_t de_ino;                // 对应的 inode 号（0 = 空闲目录项）
} DirEntry;                         // sizeof = 16 字节，每块 32 个
```

## 2. 模块总览与依赖关系

```
┌────────────────────────────────────────────────────────────┐
│  main.c         命令行解析、程序入口/退出                      │
│   │  调用 user.c / file.c / dir.c / format.c               │
└───┼────────────────────────────────────────────────────────┘
    │                          ▲
    │   user.c / file.c / dir.c（其他同学负责）
    │                          │  依赖 sb_ialloc / sb_balloc
    │                          ▼
    │               ┌──────────────────┐
    │               │  super.c         │  inode 号 & data block 号
    │               │  分配与回收       │  的分配/回收
    │               └────────┬─────────┘
    │                        │  依赖 bread / bwrite
    │                        ▼
    │               ┌──────────────────┐
    │               │  persist.c       │  块级 I/O（bread / bwrite）
    │               │  磁盘读写封装     │  封装 fseek + fread/fwrite
    │               └────────┬─────────┘
    │                        │  操作 FILE* g_fs_file
    │                        ▼
    │               ┌──────────────────┐
    │               │  filesystem.dat  │  宿主文件
    │               └──────────────────┘
    │
    ├── globals.c   全局变量定义（SuperBlock、inode 哈希表、用户表、FILE*）
    └── format.c    格式化（一次性初始化磁盘布局 + 根目录）
```

**关键依赖方向**：所有模块最终都通过 `super.c → persist.c → FILE*` 这条路径完成磁盘操作。你的模块是整条链路的底层。

## 3. globals.h / globals.c —— 全局变量

### 3.1 为什么需要这个模块

大型 C 项目中，多个 `.c` 文件需要共享同一份运行状态。如果把全局变量分散定义在各个文件中，链接器会因重复定义报错。标准做法是：

- **定义**放在 `.c` 文件中（只分配一次内存）
- **声明**放在 `.h` 文件中（`extern`，告诉其他编译单元"这个符号存在"）

```c
// globals.c — 实际分配内存
SuperBlock g_sb;                       // 全局超级块（唯一实例）
MInode *g_inode_hash[HASH_BUCKETS];    // inode 哈希表
User g_users[MAX_USERS];               // 用户数组
FILE *g_fs_file = NULL;                // 宿主文件句柄
const char *g_fs_path = NULL;          // 宿主文件路径

// globals.h — 声明给其他文件使用
extern SuperBlock g_sb;
extern MInode *g_inode_hash[HASH_BUCKETS];
extern User g_users[MAX_USERS];
extern FILE *g_fs_file;
extern const char *g_fs_path;
```

### 3.2 每个全局变量的用途

| 变量 | 类型 | 用途 |
|------|------|------|
| `g_sb` | `SuperBlock` | 整个文件系统的"状态寄存器"，含空闲资源栈和区域布局信息 |
| `g_inode_hash[]` | `MInode*[HASH_BUCKETS]` | inode 缓存哈希表，`iget(ino)` 先查这里，miss 再读磁盘 |
| `g_users[]` | `User[MAX_USERS]` | 用户数组，每个 User 有自己的打开文件表、工作目录 |
| `g_current_user_idx` | `int` | 当前登录用户在 g_users[] 中的索引，-1 = 未登录 |
| `g_fs_file` | `FILE*` | 所有 `bread/bwrite` 操作的目标文件句柄 |
| `g_fs_path` | `const char*` | 宿主文件路径字符串（如 `"filesystem.dat"`） |

### 3.3 生命周期

`g_sb`、`g_inode_hash`、`g_fs_file` 在程序启动时初始化，在退出时清理（`sb_save + fs_shutdown`）。它们的作用域是整个进程生命周期。

## 4. persist.h / persist.c —— 块 I/O 层

### 4.1 设计目的

将底层 C 标准库 I/O（`fopen/fread/fwrite/fseek/flush`）封装为文件系统语义的"块级读写"。上层代码不需要关心 `FILE*` 和字节偏移，只需操作逻辑块号。

### 4.2 接口一览

```c
int  block_init(const char *host_path);  // 以 "rb+" 模式打开宿主文件
int  bread(uint16_t blkno, void *buf);   // 读块 → buf
int  bwrite(uint16_t blkno, const void *buf); // buf → 写块
void sb_load(void);                      // 从块 0 读 g_sb
void sb_save(void);                      // g_sb 写回块 0
void fs_shutdown(void);                  // fflush + fclose + NULL
```

### 4.3 核心实现

```c
int bread(uint16_t blkno, void *buf)
{
    if (!g_fs_file)              return -1;  // 句柄未初始化
    if (blkno >= TOTAL_BLOCKS) {             // 越界保护
        memset(buf, 0, BLOCK_SIZE);          // 返回全零而非 crash
        return -1;
    }
    fseek(g_fs_file, (long)blkno * BLOCK_SIZE, SEEK_SET);
    size_t n = fread(buf, 1, BLOCK_SIZE, g_fs_file);
    if (n != BLOCK_SIZE) {                   // 读到文件末尾或 I/O 错误
        memset(buf, 0, BLOCK_SIZE);
        return -1;
    }
    return 0;
}

int bwrite(uint16_t blkno, const void *buf)
{
    // ...
    fflush(g_fs_file);   // 强制把 C 库缓冲刷到 OS，减少断电丢数据风险
    return 0;
}
```

### 4.4 错误处理策略

`bread` 在越界或读取不完整时 **不 crash、不抛异常**，而是返回 -1 并将 `buf` 清零。调用者自行检查返回值。这种设计让上层代码可以安全调用而不需要预先校验块号。

### 4.5 sb_load 和 sb_save

```c
void sb_load(void) {
    bread(SUPER_BLOCK_NO, &g_sb);   // 块 0 → g_sb
}

void sb_save(void) {
    g_sb.s_fmod = 0;                // 清脏标记
    bwrite(SUPER_BLOCK_NO, &g_sb);  // g_sb → 块 0
}
```

`s_fmod` 的作用：标记超级块是否被修改过。如果在 `s_fmod == 1` 时程序崩溃，下次启动可检测到不一致。本项目为简化起见，只在退出时检查。

## 5. super.h / super.c —— 超级块管理

这是你负责的模块中**算法最密集**的部分。

### 5.1 职责

管理两种全局资源的分配和回收：

| 资源 | 取值空间 | 策略 |
|------|---------|------|
| inode 号 | 1 ~ TOTAL_INODES (512) | 内存栈（50 槽位）+ 磁盘扫描 |
| 数据块号 | DATA_AREA_START ~ TOTAL_BLOCKS-1 (34~545) | 成组链接法（Group Linking） |

### 5.2 inode 分配：内存栈 + 磁盘扫描

**正常路径（O(1)）：**

```c
uint16_t sb_ialloc(void)
{
    if (g_sb.s_nfree == 0) {         // 栈空 → 回填
        if (scan_free_inodes() == 0)
            return 0;                // 磁盘 inode 区满
    }
    g_sb.s_nfree--;                  // 栈顶下移
    return g_sb.s_free_inode[g_sb.s_nfree];   // 弹栈（LIFO）
}
```

`s_free_inode[50]` 是内存中的一个 LIFO 栈，`s_nfree` 是栈顶指针（即当前栈内元素个数）。

**栈空时的回填（scan_free_inodes）：**

```c
static int scan_free_inodes(void)
{
    int found = 0;
    for (uint16_t ino = ROOT_INO; ino <= TOTAL_INODES; ino++) {
        if (read_inode_mode(ino) == 0) {    // di_mode == 0 → 空闲
            g_sb.s_free_inode[found++] = ino;
            if (found >= 50) break;
        }
    }
    g_sb.s_nfree = found;
    return found;
}
```

`read_inode_mode(ino)` 只读磁盘 inode 的前 2 字节（`di_mode`），避免加载整个 32 字节的 `DiskInode`。位置计算：

```c
blk = INODE_AREA_START + (ino - 1) / INODES_PER_BLOCK;
off = ((ino - 1) % INODES_PER_BLOCK) * DISK_INODE_SIZE;
```

**释放（sb_ifree）：**

```c
void sb_ifree(uint16_t ino)
{
    zero_disk_inode(ino);            // 磁盘 inode 清零（di_mode = 0）
    if (g_sb.s_nfree < 50) {
        g_sb.s_free_inode[g_sb.s_nfree++] = ino;   // push
    }
    // 栈满时不做 push — 磁盘已清零，scan 会重新发现
}
```

这是一种**降级回收**策略：栈满时不维护，依赖扫描机制后期回收。

### 5.3 数据块分配：成组链接法（Group Linking）

这是整个文件系统中最精巧的设计。

**问题背景**：`SuperBlock` 只占 512 字节，其中 `s_free_block[50]` 只能存 50 个空闲数据块号。但数据区有 512 个块。如何用 50 个槽位管理 512 个资源？

**解决方案**：利用空闲块本身来存储空闲号列表，形成隐式链表。

#### 正常分配（O(1)）：

```c
uint16_t sb_balloc(void)
{
    if (g_sb.s_nfree_block == 0) return 0;   // 磁盘满
    g_sb.s_nfree_block--;
    uint16_t blkno = g_sb.s_free_block[g_sb.s_nfree_block]; // 弹栈

    // 关键：弹出后栈空了 → blkno 是下一组的链表头
    if (g_sb.s_nfree_block == 0) {
        uint16_t header[256];
        bread(blkno, header);                      // 读该块
        uint16_t count = header[0];                // 组内块数
        for (uint16_t i = 0; i < count; i++)
            g_sb.s_free_block[i] = header[i + 1];  // 恢复栈
        g_sb.s_nfree_block = count;
    }
    return blkno;
}
```

**逻辑**：`g_sb.s_free_block[50]` 里的最后一个元素弹出后，如果栈空了，说明刚刚弹出的 `blkno` 本身是一个 **group header block**，其内容的前 51 个 `uint16_t` 就是下一个组的完整栈。这就形成了链式结构：

```
g_sb.s_free_block[50]  →  块A 的 header[1..50]  →  块B 的 header[1..50]  →  ...
         (当前组)              (下一组)                  (再下一组)
```

#### 释放（栈满时的 dump）：

```c
void sb_bfree(uint16_t blkno)
{
    if (g_sb.s_nfree_block < 50) {
        g_sb.s_free_block[g_sb.s_nfree_block++] = blkno;  // 直接 push
    } else {
        // 栈满 → 把当前 50 个块号写入 blkno 块
        uint16_t header[256];
        header[0] = 50;
        for (int i = 0; i < 50; i++)
            header[i + 1] = g_sb.s_free_block[i];
        bwrite(blkno, header);

        // 栈重置：只含 blkno 一个元素
        g_sb.s_free_block[0] = blkno;
        g_sb.s_nfree_block = 1;
    }
}
```

**这就是成组链接法的核心机制**：

- 分配时：弹栈 → 栈空则从被弹块读下一组 → 恢复栈
- 释放时：栈满则 dump 当前栈到被释放块 → 栈重置

每个空闲块在成为链表头之前被写入 header，成为链表头后被分配时会恢复里面的栈。**空间复杂度 O(1)（超级块永不增长），时间复杂度 O(1)（弹栈/压栈），任意多个空闲块都能管理。**

### 5.4 格式化辅助函数

```c
// 把数据区全部 512 个块 push 进空闲栈
void sb_init_free_blocks(uint16_t start, uint16_t count)
{
    for (uint16_t blk = start; blk < start + count; blk++)
        sb_bfree(blk);   // 逐块 push，栈满自动 dump，形成初始组
}

// 初始化空闲 inode 栈（跳过根目录 inode 1）
void sb_init_free_inodes(uint16_t total)
{
    for (uint16_t ino = 2; ino <= total; ino++)
        zero_disk_inode(ino);                       // 清零
    for (uint16_t ino = total; ino >= 2; ino--)     // 逆序 push
        g_sb.s_free_inode[g_sb.s_nfree++] = ino;
}
```

逆序 push 的原因：LIFO 栈中，后 push 的元素先弹出。逆序使小编号在栈顶，分配时优先拿到小编号 inode。

## 6. format.h / format.c —— 文件系统格式化

### 6.1 触发时机

- 宿主文件不存在时（`block_init` 返回 -1）→ `main.c` 自动调用
- 用户手动输入 `format` 命令

### 6.2 10 步流程

```
1. fopen(g_fs_path, "wb+")             创建/截断宿主文件
2. for (i = 0; i < TOTAL_BLOCKS; i++)  所有块写零
      fwrite(zero, 1, BLOCK_SIZE, g_fs_file)
3. memset(&g_sb, 0, sizeof(g_sb))      构造超级块
   g_sb.s_inode_area_start = INODE_AREA_START
   g_sb.s_inode_area_size  = INODE_AREA_BLOCKS
   g_sb.s_data_area_start  = DATA_AREA_START
   g_sb.s_data_area_size   = DATA_AREA_BLOCKS
4. sb_init_free_inodes(TOTAL_INODES)   初始化空闲 inode 栈
5. sb_init_free_blocks(...)            初始化空闲 data block 栈
6. (根目录用 inode 1，不通过 sb_ialloc)
7. uint16_t root_blk = sb_balloc()     为根目录分配一个数据块
8. DiskInode root_di = {               构造根目录 inode
       .di_mode  = DEFAULT_DIR_MODE,    // FT_DIR | rwxr-xr-x
       .di_nlink = 2,                   // "." 和 ".."
       .di_uid   = 0,
       .di_size  = 2 * DIR_ENTRY_SIZE,  // 32 字节
       .di_addr[0] = root_blk
   }
   write_disk_inode(ROOT_INO, &root_di)
9. DirEntry dir_block[32] = {{".", ROOT_INO}, {"..", ROOT_INO}}
   bwrite(root_blk, dir_block)         写入 "." 和 ".."
10. sb_save()                          超级块写回块 0
```

### 6.3 write_disk_inode 的实现

```c
static void write_disk_inode(uint16_t ino, const DiskInode *dip)
{
    // 计算 inode 在 inode 区的物理位置
    uint16_t blk = INODE_AREA_START + (ino - 1) / INODES_PER_BLOCK;
    uint16_t off = ((ino - 1) % INODES_PER_BLOCK) * DISK_INODE_SIZE;

    char buf[BLOCK_SIZE];
    bread(blk, buf);                         // 读整块
    memcpy(buf + off, dip, DISK_INODE_SIZE); // 覆盖目标 inode
    bwrite(blk, buf);                        // 写回
}
```

注意这里用的是 **read-modify-write** 模式：同一块中可能有 16 个 inode，只覆盖其中一个，其余保持不变。

### 6.4 根目录的特殊性

根目录使用了 `inode 1` 和 `一个数据块`，但这两者**不通过 sb_ialloc/sb_balloc 获取**，而是直接赋值。原因是：

1. 格式化过程中 `sb_init_free_*` 把 inode 1 也放进了空闲栈
2. 如果之后用 `sb_ialloc()` 随机分配，inode 1 可能被普通文件占用
3. 根目录需要一个**确定的** inode 号（`ROOT_INO = 1`），便于系统通过 `/` 快速访问

## 7. main.c —— 主控模块

### 7.1 程序三阶段

**阶段 I：启动**

```c
int main(int argc, char *argv[])
{
    const char *fs_path = (argc > 1) ? argv[1] : "filesystem.dat";

    if (block_init(fs_path) == 0) {
        // 宿主文件存在 → 加载已有系统
        sb_load();               // 块 0 → g_sb
        inode_hash_init();       // 初始化 g_inode_hash[] 空表
        user_init();             // 初始化 g_users[], 创建 /usr1~/usr8
    } else {
        // 宿主文件不存在 → 格式化新系统
        g_fs_path = fs_path;     // 必须在 fs_format() 之前赋值
        inode_hash_init();
        if (fs_format() != 0)
            return 1;            // 格式化失败
        user_init();
    }
```

**阶段 II：REPL 命令循环**

```c
    char line[1024];
    for (;;) {
        printf("$ "); fflush(stdout);
        if (!fgets(line, sizeof(line), stdin)) break;  // EOF

        line[strcspn(line, "\n")] = '\0';  // 去除换行符
        char *s = line;
        char *cmd = next_arg(&s);          // 第一个 token 是命令名

        // 命令分发（if-else if 链，因为 C 的 switch 不支持字符串）
        if (strcmp(cmd, "exit") == 0 || strcmp(cmd, "quit") == 0)
            break;
        else if (strcmp(cmd, "format") == 0)
            { inode_flush_all(); fs_shutdown(); fs_format(); user_init(); }
        else if (strcmp(cmd, "login") == 0)
            { fs_login(next_arg(&s), next_arg(&s) ?: ""); }
        // ... 其余命令 ...
    }
```

**阶段 III：安全退出**

```c
    inode_flush_all();  // 所有脏 inode 写回磁盘（遍历哈希表，I_DIRTY 的写回）
    sb_save();          // g_sb → 块 0（持久化空闲栈和区域信息）
    fs_shutdown();      // fflush + fclose + g_fs_file = NULL
    return 0;
}
```

### 7.2 命令解析辅助函数

**next_arg：原地修改字符串的分词器**

```c
static char *next_arg(char **s)
{
    while (**s == ' ' || **s == '\t') (*s)++;   // 跳过前导空白
    if (**s == '\0') return NULL;               // 到达串尾

    char *start = *s;                           // token 起点
    while (**s && **s != ' ' && **s != '\t')    // 扫描到分隔符
        (*s)++;
    if (**s) { **s = '\0'; (*s)++; }            // 截断 + 指针后移
    return start;
}
```

这是经典的 **strtok 替代方案**，利用原始字符串中的空格写入 `\0` 来截断 token。注意：这会破坏原始缓冲区内容，不能用于 `const` 字符串。

**parse_mode：字符串 → 打开标志位**

```c
static uint16_t parse_mode(const char *s) {
    if (!s) return 0;
    if (!strcmp(s, "r")  || !strcmp(s, "read"))   return O_RDONLY;
    if (!strcmp(s, "w")  || !strcmp(s, "write"))  return O_WRONLY | O_TRUNC;
    if (!strcmp(s, "a")  || !strcmp(s, "append")) return O_WRONLY | O_APPEND;
    if (!strcmp(s, "rw") || !strcmp(s, "wr"))     return O_RDWR;
    return 0;  // 无法识别 → 调用者默认用 O_RDONLY
}
```

**parse_fd：字符串 → fd 编号（带校验）**

```c
static int parse_fd(const char *s) {
    if (!s || !*s) return -1;
    for (const char *p = s; *p; p++)
        if (*p < '0' || *p > '9') return -1;  // 含有非数字字符
    int fd = 0;
    for (const char *p = s; *p; p++)
        fd = fd * 10 + (*p - '0');
    if (fd >= MAX_OPEN_FILES) return -1;      // 超出有效范围
    return fd;
}
```

## 8. 完整调用链示例

以用户输入 `create /a.txt rw` 为例的完整调用链路：

```
main.c: next_arg 切分 → cmd="create", path="/a.txt", mode="rw"
         ↓
file.c: fs_create("/a.txt", mode)
         ↓  调用 namei("/a.txt") → 路径解析失败（文件不存在）
         ↓  调用 sb_ialloc()
         ↓      ↓
super.c: sb_ialloc()
         ↓      g_sb.s_nfree-- → 弹出 s_free_inode[s_nfree]
         ↓      返回 ino = 42（假设分配到了 42 号 inode）
         ↓
         ↓  调用 sb_balloc()
         ↓      ↓
super.c: sb_balloc()
         ↓      g_sb.s_nfree_block-- → 弹出一个块号
         ↓      返回 data_blk = 87（假设分配到了 87 号块）
         ↓
         ↓  构造 DiskInode（di_mode=DEFAULT_FILE_MODE, di_addr[0]=87）
         ↓  调用 write_disk_inode → format.c 内的辅助函数
         ↓      ↓
         ↓      bread(blk, buf) → memcpy → bwrite(blk, buf)
         ↓           ↓
persist.c: bread(blk, buf)
         ↓      fseek(g_fs_file, blk * 512, SEEK_SET)
         ↓      fread(buf, 1, 512, g_fs_file)
         ↓      返回 0
         ↓
         ↓  调用 dir_add_entry()  把 "a.txt" 写入父目录的 DirEntry 数组
         ↓      ↓
         ↓      bread(parent_data_blk, buf) → 找到空闲 de_ino==0 的位置
         ↓      → 写入 de_name="a.txt", de_ino=42
         ↓      → bwrite(parent_data_blk, buf)
         ↓
         ↓  返回文件描述符 fd 给 main.c
         ↓
main.c: 打印 "Created file: /a.txt (fd=3)"
```

每一步都经过 **super.c（资源分配）→ persist.c（磁盘读写）** 这条路径。没有 `bread/bwrite`，所有 inode 和目录的操作都无法持久化；没有 `sb_ialloc/sb_balloc`，就无法创建任何新文件。
