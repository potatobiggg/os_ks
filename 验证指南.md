# VFS 文件系统验证指南

## 编译

```bash
cd f:/osks/vfs
gcc -std=c11 -g -O0 -o vfs.exe src/*.c
```

## 启动

```bash
./vfs.exe mytest.dat
```

首次启动自动格式化，后续启动自动加载已有文件系统。

---

## 逐一验证 12 条命令

### 1. format — 格式化文件系统

```
format
```

**预期输出：**
```
Formatting file system...
File system formatted: 546 blocks, 512 inodes
Data area: blocks 34-545 (512 blocks)
```

> PDF 第 41 条要求。首次启动也会自动格式化。

---

### 2. login / logout — 多用户登录与注销

PDF 要求支持 usr1 ~ usr8 共 8 个用户。

```
login usr1
```

**预期输出：** `Welcome, usr1! Current directory: /`

```
dir
```

**预期输出：** 根目录列表（只有 `.` 和 `..`）

```
logout
```

**预期输出：** `Goodbye, usr1!`

---

### 3. mkdir — 创建目录

```
login usr1
mkdir /home
mkdir /home/usr1
mkdir /docs
mkdir /docs/manual
dir /
```

**预期输出：** 根目录显示 `home` 和 `docs` 两个目录

```
dir /docs
```

**预期输出：** `/docs` 下显示 `manual` 子目录

> 验证 PDF 要求的"多级目录：可有多级子目录"

---

### 4. chdir — 改变当前目录

```
chdir /home/usr1
```

**预期输出：** `chdir: current directory is '/home/usr1'`

```
create test.txt
dir
```

**预期输出：** 显示在 `/home/usr1` 下创建了 `test.txt`

```
chdir /
dir
```

**预期输出：** 回到根目录

---

### 5. dir — 列出文件目录

```
dir /home
```

**预期输出：** 显示 `/home` 下的 `usr1` 目录

```
dir /home/usr1
```

**预期输出：** 显示 `test.txt` 文件，含 inode 号、大小、类型

---

### 6. create — 创建文件

```
create /data.txt
create /home/usr1/note.txt
dir /
dir /home/usr1
```

**预期输出：** 根目录出现 `data.txt`，`/home/usr1` 出现 `note.txt`

---

### 7. open / close — 打开与关闭文件

```
open /data.txt w
```

**预期输出：** `open: fd=0, file='/data.txt', size=0 bytes`

```
close 0
```

**预期输出：** `close: fd=0 closed`

---

### 8. write / read — 写与读文件

```
open /data.txt w
write 0 Hello, this is a test message for VFS!
close 0
open /data.txt r
read 0 100
close 0
```

**预期输出：**
```
write: wrote 39 bytes
open: fd=0, file='/data.txt', size=39 bytes
Read 39 bytes: "Hello, this is a test message for VFS!"
```

---

### 9. delete — 删除文件

```
delete /data.txt
dir /
```

**预期输出：** `data.txt` 从根目录消失

**验证不可删除非空目录：**

```
mkdir /testdir
create /testdir/a.txt
delete /testdir
```

**预期输出：** `delete: directory 'testdir' is not empty`

---

### 10. 多用户隔离验证

PDF 要求的用户访问控制：

```
logout
login usr2
create /home/usr2/private.txt
open /home/usr2/private.txt w
write 0 Secret data of usr2
close 0
logout
login usr1
delete /home/usr2/private.txt
```

**预期输出：** `delete: permission denied`

> usr1 无法删除 usr2 的文件，验证了用户隔离

```
dir /home/usr2
```

**预期输出：** `private.txt` 仍然存在（未被删除）

---

### 11. 持久化验证

PDF 第 10 页要求："模拟的文件系统可以保存，以便下次开机时再用"

```
logout
exit
```

**预期输出：** `Saving and exiting...` → `Goodbye.`

**重新启动：**

```bash
./vfs.exe mytest.dat
```

```
login usr1
dir /home/usr1
```

**预期输出：** 之前创建的 `note.txt` 仍然存在

```
open /home/usr1/note.txt r
read 0 50
close 0
```

**预期输出：** 能读出之前写入的内容

---

### 12. 多级嵌套目录 + 深层文件

验证混合索引和多级目录：

```
login usr1
mkdir /a
mkdir /a/b
mkdir /a/b/c
mkdir /a/b/c/d
chdir /a/b/c/d
create deep.txt
open deep.txt w
write 0 This file is 4 levels deep!
close 0
chdir /
dir /a/b/c/d
```

**预期输出：** 显示 `deep.txt` 文件

## 一键验证脚本

将以下内容保存为 `test_commands.txt`：

```
login usr1
format
mkdir /home
mkdir /home/usr1
mkdir /docs
chdir /home/usr1
create note.txt
open note.txt w
write 0 Hello VFS!
close 0
open note.txt r
read 0 50
close 0
create data.bin
dir
delete data.bin
dir
chdir /
dir /home/usr1
mkdir /a
mkdir /a/b
mkdir /a/b/c
chdir /a/b/c
create nested.txt
dir
chdir /
delete /home/usr1
logout
login usr2
create /home/usr1/file.txt
logout
login usr1
delete /docs
dir /
logout
exit
```

运行：

```bash
./vfs.exe test.dat < test_commands.txt
```

## 验证清单对照 PDF

| PDF 要求 | 对应命令 | 验证方式 |
|----------|---------|---------|
| 多用户 usr1~usr8 | `login usr1` ~ `login usr8` | 不同用户登录成功 |
| 多级目录 | `mkdir /a/b/c` | 嵌套创建成功 |
| login/logout | `login` / `logout` | 登录后显示用户名，退出后操作被拒绝 |
| create | `create /path` | 文件出现在目录列表中 |
| open/close | `open /path r` / `close 0` | 返回 fd，关闭后 fd 释放 |
| read/write | `write 0 text` / `read 0 N` | 读回的数据与写入一致 |
| delete | `delete /path` | 文件从目录消失 |
| mkdir/chdir/dir | 见上 | 目录操作正确 |
| format | `format` | 重建文件系统 |
| 持久化 | `exit` 后重新启动 | 数据保留 |
| 用户隔离 | usr1 删 usr2 文件 | 权限拒绝 |
| 命令行交互 | `$` 提示符 | 逐条命令执行 |

## 排查问题

如果持久化后 `read` 返回 0 字节，请检查：

1. 两次启动是否使用了**相同的 .dat 文件路径**
2. 是否在 exit 前误执行了 `format`（format 会清空所有数据）
3. 在第二次启动后，`open` 命令显示的 `size` 是否正确。如果 size 正确但 read 返回 0，说明 inode 的 di_addr[0] 丢失，请贴出终端完整输出以便排查。
